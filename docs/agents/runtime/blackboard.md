### createIssueLink direction — VERIFY by read-back (and against the UI)
- To create "X is blocked by Y" (Y blocks X): `type:"Blocks"`, `inwardIssue: Y` (the **blocker**), `outwardIssue: X` (the **blocked**). On read-back, the blocker shows as **`inwardIssue`** ("is blocked by") on the blocked issue; things it blocks show as `outwardIssue` ("blocks"). So **a task's blockers = its `inwardIssue` links.** ⚠️ An earlier version of this note had it reversed (said blocker = `outwardIssue`) — it cost a whole Epic-2 DAG inversion + 10 links a human had to delete by hand. Always read back one link **and** check the UI "blocks / is blocked by" heading.

## Tooling hazards

### 2026-06-21 21:25 agent-D — ⚠ Agents share ONE git clone — work in your own `git worktree` (TM-124)
- **Hazard:** all agents operate in the same working clone (`~/Projects/teammarhaba-repo`). A clone has a single HEAD/working tree, so if another agent runs `git checkout` between your `git checkout -b feature/…` and your `git commit`, **your commit lands on whatever branch HEAD is on at commit time** (it landed on `main` for me on TM-108), the feature branch stays empty, the push is empty, and `gh pr create` fails with *"No commits between main and feature/…"*. The commit is only recoverable via `git reflog` — one prune from lost.
- **Also:** a stray commit on local `main` leaves it ahead of `origin/main`, so the next agent's `git checkout main && git pull && checkout -b` either inherits your commit or hits a divergent-history merge.
- **Fix (do this):** give yourself an isolated working tree — `git worktree add -b <type>/TM-XX-desc /tmp/<agentId>-wt origin/main`, do ALL your edits/commits/rebases there. A worktree has its own HEAD that other agents in the main dir can't move. Cheap (shares the object store). Remove with `git worktree remove`. I rebuilt TM-108 + this note this way.
- **If you already committed to the wrong branch:** `git branch -f <your-feature> <sha>` (from reflog) then push; reset local `main` only when it's checked out + clean (`[ "$(git branch --show-current)" = main ] && [ -z "$(git status --porcelain)" ] && git reset --hard origin/main`).
- **Before every `commit`/`push`:** sanity-check `git branch --show-current` is YOUR branch. Full write-up + proposed protocol fix: **TM-124**.

## RBAC

### 2026-06-22 agent-A — Role claim → authority wiring + first-admin bootstrap (TM-110)
- **Authorization source of truth = the Firebase `role` custom claim**, not the DB. `auth.RoleClaims` centralises it: `CLAIM="role"`, `roleFrom(claims)` (lenient, **fail-safe to USER** on absent/blank/unknown/null), `authorities(role)` → `ROLE_USER`/`ROLE_ADMIN`. `FirebaseAuthenticationFilter` now grants these on the `Authentication`. So **TM-111 can just use `@PreAuthorize("hasRole('ADMIN')")`** — the authority is already there. The `users.role` column mirrors the claim for reporting only.
- **Set a role** via `auth.RoleService.assignRole(uid, Role)` (Admin SDK `setCustomUserClaims`). Two gotchas baked into its javadoc: (1) it **replaces** the whole custom-claims object (role is the only claim today — fine; revisit if more claims land); (2) a role change only reaches the client **after the ID token refreshes** (hourly, or force-refresh) — existing tokens keep the old claim.
- **First-admin bootstrap (chicken-and-egg fix):** `auth.AdminBootstrap` (ApplicationRunner) promotes `app.admin.bootstrap-email` (env `ADMIN_BOOTSTRAP_EMAIL`, optional) to ADMIN on startup. **Env-driven, no hard-coded identity** (replay-safe). Idempotent; best-effort (logs WARN, never fails startup) if the account hasn't signed in yet.
- **Keeps boot ADC-free:** like the verify path, `RoleService`/`AdminBootstrap` resolve `FirebaseAuth` via `ObjectProvider`, and the bootstrap **returns early when the email is blank** (dev/test/CI default) — so no Firebase/ADC init at boot. Don't make bootstrap eager or it'll break the credential-free test boot.
- All 4 new classes covered by **fast Mockito unit tests** (no Testcontainers): `RoleClaimsTest`, `FirebaseAuthenticationFilterTest`, `RoleServiceTest`, `AdminBootstrapTest`. The existing `FirebaseAuthIntegrationTest` still passes (mock token's `getClaims()` is null → USER).

### 2026-07-03 agent-tm424 — Clock-driven IT flake: Postgres TIMESTAMPTZ micro-rounding vs exact time boundaries (TM-419)
- **Symptom:** `WaitlistOfferCascadeIntegrationTest` (the offer cascade) flaked ONLY on CI — two "widen after 5 minutes" tests failed `expected: 1 but was: 0`, a different method each run, "lucky pass" on some PRs / red on main. Passed 100% locally on macOS. Prior attempts (full FK-safe DB wipe, then `@Disabled`) chased *test isolation*; that was a real-but-secondary issue.
- **Actual root cause = timestamp precision, not isolation.** Any timestamp you persist (`offer_notified_at`, etc.) is a Postgres **`TIMESTAMPTZ` = microsecond** precision, and **PG rounds on store**. `Instant.now()` is **nanosecond**-precision on Linux/CI but only **microsecond** on macOS — so a stamp written from a nanosecond clock can round **up** a few ns. Code that then does integer-division time-bucket maths *exactly on a boundary* (`Duration.between(persistedStamp, now).toMillis() / intervalMs`) reads `elapsed = 0` instead of `1`. ~50/50 → order-dependent flake invisible on a Mac.
- **Reproduce on a micros-clock host:** set the test clock to `Instant.now().plusNanos(600)` → the boundary assertion fails exactly like CI. Great A/B harness for this class of bug.
- **Fix (test-side, product left alone):** drive controlled-`Clock` tests from a **millisecond-truncated** now — `clock.setTo(Instant.now().truncatedTo(ChronoUnit.MILLIS))` — so every stamp round-trips losslessly and boundary maths are exact on every platform. Production is fine: real wall-clock advances past the boundary continuously; only exact-boundary tests care. **Rule of thumb:** never assert a persisted-timestamp time-bucket *exactly* on its boundary — truncate the driving clock to ms (or step comfortably past the boundary). Applies to any future `@Scheduled`/clock-seam IT (reminders, cascades, cutoffs).
